
# 🪨 ScoriaDB – Встраиваемая транзакционная KV-база данных на Go (MVP)

**Название кейса:** ScoriaDB

---

## 📁 Раздел 1. Разбор ТЗ

### 1.1 Проблематика
Разработчики инфраструктурного ПО постоянно ищут лёгкую встраиваемую БД, которая одновременно даёт высокую скорость (LSM), транзакционность (ACID) и возможность работы с большими значениями без раздувания памяти. Существующие решения (BoltDB – без MVCC, BadgerDB – без транзакций с изоляцией снимков, LevelDB – однопоточная) идут на компромиссы. ScoriaDB закрывает этот пробел, предлагая единый Go‑модуль, в котором объединены LSM‑дерево, WiscKey‑подобный Value Log, MVCC, оптимистичные транзакции и удобные интерфейсы (CLI, gRPC, Web UI).

### 1.2 Цель заказчика
Создать эталонный pet‑проект для демонстрации экспертизы в системном программировании, распределённых системах и построении инструментов для разработчиков. Результат должен быть готов к показу на собеседованиях и хакатонах, подниматься одной командой `docker-compose up` и предоставлять все заявленные возможности через CLI, gRPC и браузер.

### 1.3 Интерпретация пунктов ТЗ
Ниже каждый пункт ТЗ разобран по трём составляющим.

**Пункт ТЗ:** «LSM-дерево с MemTable, Value Log и WAL»  
**Реальная интерпретация:** Ядром является локальный движок с отсортированной MemTable (btree), которая периодически сбрасывается в неизменяемые SSTable на диске. Значения длиннее 64 байт выносятся в отдельный Value Log (append-only файл), а в LSM хранятся только указатели на них, чтобы не раздувать дерево. Все операции сначала фиксируются в WAL для персистентности.  
**Скрытое требование:** Необходимы блочный индекс и фильтр Блума в SSTable для быстрого поиска, а также фоновый сборщик мусора Value Log, удаляющий устаревшие записи, на которые больше не ссылаются указатели.

**Пункт ТЗ:** «MVCC и Snapshot Isolation»  
**Реальная интерпретация:** Каждая версия ключа помечается инвертированной временной меткой (timestamp = MaxUint64 – commitTS), чтобы при сортировке новые версии шли первыми. Чтение с заданным `snapshotTS` видит последнюю версию с `commitTS ≤ snapshotTS`. Удаление реализуется как запись tombstone‑версии.  
**Скрытое требование:** Необходим фоновый сборщик мусора старых версий (GC) с учётом минимального активного снапшота, чтобы диск не переполнялся.

**Пункт ТЗ:** «Транзакции: WriteBatch и интерактивные»  
**Реальная интерпретация:** `WriteBatch` – атомарное применение набора операций (аналог RocksDB). Интерактивная транзакция начинается с `Begin()`, все чтения внутри неё используют `snapshotTS = startTS`, а изменения буферизуются локально; при `Commit()` проверяется конфликт записи (оптимистичная блокировка) и все изменения фиксируются одной атомарной пачкой.  
**Скрытое требование:** При конфликте транзакцию можно повторить. Откат (`Rollback`) просто отбрасывает буфер. Механизм должен гарантировать, что в случае сбоя во время коммита не останется частично применённых данных.

**Пункт ТЗ:** «Column Families»  
**Реальная интерпретация:** Логическое разделение хранилища на несколько независимых LSM‑деревьев с собственными MemTable, SSTable и настройками компакшена.  
**Скрытое требование:** Атомарность при записи в несколько Column Families – операции внутри одного `WriteBatch` могут затрагивать разные CF и фиксируются вместе.

**Пункт ТЗ:** «gRPC API и CLI‑клиент»  
**Реальная интерпретация:** Удалённый доступ к БД через gRPC. CLI‑клиент (`scoria`) оборачивает все операции в удобные команды и поддерживает конфигурационные файлы.  
**Скрытое требование:** gRPC должен поддерживать стриминг для операции `Scan` (потоковая выдача ключей). CLI должен поддерживать автодополнение, историю команд и понятный вывод.

**Пункт ТЗ:** «Базовый Web UI»  
**Реальная интерпретация:** Веб‑интерфейс, позволяющий просматривать данные, выполнять простые запросы и видеть состояние сервера.  
**Скрытое требование:** Интерфейс должен быть одностраничным приложением (React), загружаемым как статика с сервера, и взаимодействовать через REST API.

**Пункт ТЗ:** «Аутентификация и авторизация»  
**Реальная интерпретация:** Встроенная система пользователей с хранением учётных данных в самой БД. Роли: `admin` (полный доступ), `readwrite`, `readonly`. Аутентификация по JWT-токену или API‑ключу.  
**Скрытое требование:** Все gRPC и REST‑запросы должны проверять наличие и валидность токена, а также соответствие роли требуемому действию.

### 1.4 Что можно улучшить (в следующих итерациях)
- Распределённый режим (Raft, шардирование, распределённые транзакции).
- Time Travel Queries и исторический браузер в Web UI.
- Row‑Level Security (RLS) и политики доступа на уровне ключей.
- Git‑подобное ветвление данных (branching).
- Интеграция с Kubernetes (Helm‑чарт, оператор).

## 📁 Раздел 2. Идеи для фичей

### 2.1 Обязательный функционал (MVP)
1. **LSM‑движок** с MemTable (btree), Value Log (append‑only, mmap), WAL (CRC32), SSTable (блочный индекс, префиксное сжатие, Bloom‑фильтр), Leveled Compaction, сжатие Snappy/Zstd.
2. **MVCC** с инвертированными метками времени, Snapshot Isolation, tombstone.
3. **WriteBatch** – атомарное применение пачки операций.
4. **Интерактивные транзакции** – Begin, Get (со снапшотом), Put/Delete, Commit (оптимистичная блокировка), Rollback.
5. **Column Families** – отдельные деревья с настройками, атомарность через WriteBatch.
6. **Embedded Go API** – интерфейс `DB` с методами для прямой работы в коде.
7. **gRPC API** – операции Get, Set, Delete, Scan (server‑streaming), транзакции.
8. **CLI‑клиент** (`scoria`) – команды kv, txn, admin, конфигурация, автодополнение.
9. **Web UI** – React SPA, страницы просмотра данных и статуса сервера, базовая панель управления пользователями.
10. **Аутентификация/авторизация** – пользователи, bcrypt‑хеши, роли, JWT‑токены.
11. **Прометеус‑метрики** (`/metrics`), health‑check (`/health`, `/ready`).
12. **Контейнеризация** – multi‑stage Dockerfile, docker‑compose с одной нодой.

### 2.2 Необязательный функционал (выходит за рамки MVP, запланирован на будущее)
- Raft‑репликация, шардирование, распределённые транзакции.
- Time Travel Queries, визуальный слайдер истории в Web UI.
- Compaction Visualizer (живая визуализация слияния файлов).
- Row‑Level Security, аудит, Git‑подобное ветвление данных.

### 2.3 Киллер‑фичи (5 штук)

🎯 **Название фичи:** Zero‑copy Value Log с гибридным доступом (WiscKey + mmap)  
🏷️ **Категория:** ⚡ Производительность  
❓ **Контекст:** Большие значения (например, json‑документы) не должны раздувать LSM‑дерево и требовать лишнего копирования при чтении.  
💎 **Почему киллер:** Значения хранятся в отдельном файле, отображённом в память через mmap, что позволяет читать их напрямую без выделения дополнительной памяти и десериализации. Это даёт скорость, сравнимую с LMDB, и экономит оперативную память по сравнению с классическими LSM‑движками.  
⚙️ **Как реализовать:** При `Put` значение длиннее порога записывается в конец VLog, в MemTable сохраняется только указатель (смещение и размер). Файл VLog открывается через `syscall.Mmap`, и при чтении возвращается слайс, указывающий прямо в отображённую область. Для безопасности пересчитывается CRC32 значения, хранящийся перед ним в VLog.

🎯 **Название фичи:** Транзакции с оптимистичной блокировкой на основе временных меток  
🏷️ **Категория:** 🏗️ Архитектура  
❓ **Контекст:** Интерактивные транзакции с конкурентным доступом должны гарантировать изоляцию без тяжёлых блокировок.  
💎 **Почему киллер:** Разработчик может выполнять сложные операции (например, перевод между счетами) с гарантией ACID, не блокируя других читателей. При конфликте транзакция откатывается, и клиент может её повторить.  
⚙️ **Как реализовать:** При `Commit` для каждого изменённого ключа проверяется, нет ли в движке версии с `commitTS > startTS`. Если такая есть – возвращается ошибка конфликта, и транзакция не применяется. В противном случае все новые версии записываются с одним `commitTS`.

🎯 **Название фичи:** CLI‑клиент с интерактивным режимом и AI‑ассистентом  
🏷️ **Категория:** 🤖 AI/ML (и 🎨 UI/UX)  
❓ **Контекст:** Разработчики ценят удобные консольные инструменты. Возможность запрашивать данные на естественном языке делает работу с БД ещё быстрее.  
💎 **Почему киллер:** Вместо запоминания точных ключей можно написать «покажи всех пользователей за вчера», и CLI сам преобразует запрос в команду `Scan`.  
⚙️ **Как реализовать:** Команда `scoria ask "<текст>"` отправляет запрос в локальный Ollama (или другой LLM) с системным промптом, требующим вернуть JSON вида `{ "command": "SCAN", "args": {"prefix": "..."} }`. После парсинга JSON выполняется соответствующий gRPC‑вызов.

*(Примечание: в MVP AI‑ассистент опционален, может быть продемонстрирован на стенде.)*

🎯 **Название фичи:** Web UI с просмотром данных и консолью администратора  
🏷️ **Категория:** 🎨 UI/UX  
❓ **Контекст:** Даже встраиваемая БД должна быть удобна в отладке и администрировании.  
💎 **Почему киллер:** Пользователь в браузере может посмотреть ключи, выполнить простые команды, управлять пользователями, не открывая терминал.  
⚙️ **Как реализовать:** React‑приложение, встроенное в бинарник через `embed.FS`. Использует REST API (обёртка над gRPC‑backend) для запросов и WebSocket для нотификаций о событиях (опционально).

🎯 **Название фичи:** Column Families с атомарным WriteBatch  
🏷️ **Категория:** 🏗️ Архитектура  
❓ **Контекст:** Разделение данных на логические домены с разными политиками хранения требует отдельных пространств ключей.  
💎 **Почему киллер:** Одно физическое хранилище может обслуживать, например, сессии (с TTL) и постоянные данные профиля, причём операции, затрагивающие оба CF, будут атомарны.  
⚙️ **Как реализовать:** Каждое CF – отдельный экземпляр LSM‑движка. `WriteBatch` содержит список пар `{CF, Key, Value}`. При `ApplyBatch` все операции фиксируются вместе внутри одной транзакции на уровне WAL, затем разносятся по соответствующим MemTable.

### 2.4 Идеи для безопасности (выбрано 4 согласно Правилу 8 для инструментов разработчика)
1. **JWT‑аутентификация** – все gRPC‑ и REST‑запросы к API (кроме health‑check) требуют заголовок `Authorization: Bearer <token>`.
2. **Хеширование паролей bcrypt** – учётные данные пользователей хранятся с солью, что предотвращает утечку в случае компрометации.
3. **Ролевая модель** – предопределённые роли `admin`, `readwrite`, `readonly`; проверка прав на уровне middleware.
4. **Rate Limiting (опционально)** – middleware ограничивает количество запросов от одного клиента, защищая от DoS.

### 2.5 Идеи для кастома (согласно Правилу 6 – настройки внешнего вида и поведения)
- Конфигурируемые параметры LSM: размер MemTable, количество уровней, стратегия компакшена.
- Выбор алгоритма сжатия (Snappy, Zstd).
- Тема оформления CLI и Web UI (светлая/тёмная).
- Уровень детализации логов.
- Собственные метрики Prometheus.

### 2.6 Особенности фронтенда
- **Адаптивность:** Web UI построен на Tailwind CSS, корректно отображается на планшетах и настольных ПК.
- **PWA:** Веб‑интерфейс может быть установлен как приложение (Service Worker кэширует статику), доступен офлайн‑просмотр закэшированных данных.
- **WebSocket:** Используется для получения мгновенных уведомлений об изменении данных и статусе сервера.
- **Функциональное равенство с CLI:** Согласно Правилу 7, все операции, доступные в CLI (Get, Set, Delete, Scan, управление пользователями), продублированы в Web UI через соответствующие формы.

## 📁 Раздел 3. Реализация бэкенда

### 3.1 Общая архитектура
**Классификация кейса:** Категория 6 – Инструменты для разработчиков / CLI‑утилиты (с пограничными элементами Категории 5 – внутренние инструменты, дающие аудит и визуализацию).  
**Стек технологий (согласно Правилу 4 для Категории 6):**
- Язык: Go 1.23+
- CLI: Cobra + Viper
- gRPC: google.golang.org/grpc

- Хранение данных: собственный LSM‑движок, Value Log, WAL, SSTable
- Контейнеризация: Docker, docker‑compose

**Архитектурный стиль:** Модульный монолит – основной бинарник сервера содержит встроенный движок, gRPC‑сервер, HTTP‑сервер (для Web UI и REST-прокси), CLI подключается к серверу по gRPC.  
**Структура папок (монорепозиторий):**

### 3.2 Компоненты бэкенда (основные)
- **Движок хранения (`engine`):** реализует `DB` интерфейс, управляет MemTable, SSTable, VLog, компакшеном.
- **MVCC‑слой:** кодирование ключей с инвертированной меткой, разрешение версий при чтении.
- **Транзакционный менеджер (`txn`):** координация WriteBatch и интерактивных транзакций.
- **Column Families (`cf`):** реестр именованных движков, маршрутизация операций.
- **gRPC‑сервер:** реализует эндпоинты для внешнего доступа.
- **REST API:** тонкая прослойка над gRPC для Web UI.
- **Сервис аутентификации (`auth`):** управление пользователями, хеширование, выпуск JWT.
- **Health‑сервер:** эндпоинты `/health`, `/ready`, `/metrics`.

### 3.3 API и протоколы
- **gRPC** – основной протокол общения с сервером (поддержка стриминга для `Scan`).
- **REST** – дублирует gRPC‑функции для Web UI.
- **WebSocket** – для push‑уведомлений (изменения ключей, статус).
- **Встраиваемый интерфейс** – `pkg/scoria.DB` для использования библиотеки в коде без сети.

### 3.3.1 Перечень основных эндпоинтов (gRPC / REST)

**gRPC‑сервис `ScoriaDB`:**
- `Get(GetRequest)` → `GetResponse`  
  Запрос: `key`, `snapshot_ts`. Ответ: `value`, `found`, `error`.
- `Put(PutRequest)` → `PutResponse`  
  Запрос: `key`, `value`, `cf_name` (опционально). Ответ: `commit_ts`, `error`.
- `Delete(DeleteRequest)` → `DeleteResponse`  
  Запрос: `key`, `cf_name`. Ответ: `commit_ts`.
- `Scan(ScanRequest)` → stream `ScanResponse`  
  Запрос: `prefix`, `cf_name`. Стрим возвращает записи `key`, `value`.
- `BeginTxn(BeginRequest)` → `BeginResponse`  
  Запрос: пустой. Ответ: `txn_id`, `start_ts`.
- `CommitTxn(CommitRequest)` → `CommitResponse`  
  Запрос: `txn_id`, список `{op_type, key, value}`. Ответ: `error`.
- `RollbackTxn(RollbackRequest)` → `RollbackResponse`  
  Запрос: `txn_id`.
- `CreateUser(CreateUserRequest)` → `CreateUserResponse`
- `Authenticate(AuthRequest)` → `AuthResponse`

**REST‑эндпоинты (например, для Web UI):**
- `GET /api/v1/kv/{key}` – получить значение.
- `PUT /api/v1/kv/{key}` – записать.
- `DELETE /api/v1/kv/{key}` – удалить.
- `POST /api/v1/kv/scan` – сканировать.
- `POST /api/v1/auth/login` – получить JWT.
- `GET /api/v1/admin/users` – список пользователей (для админа).
- `GET /health`, `GET /ready`, `GET /metrics`.

### 3.4 Базы данных (внутренние Column Families)
ScoriaDB не использует внешнюю СУБД. Все данные, включая системные, хранятся в отдельных Column Families в самом движке:

- `__keyspace__` – пользовательские ключи (MVCC).
- `__auth__` – пользователи и их роли (ключ: `user:<username>`, значение: `JSON{passwordHash, apiKeyHash, roles}`).
- `__index__` – (задел на будущее) вторичные индексы.
- `__meta__` – конфигурации CF.

Индексы для ускорения поиска по `__auth__` не требуются, т.к. ключ – это username.

### 3.5 Сценарий работы
**Успешный сценарий (запись через CLI):**
1. Пользователь выполняет `scoria set user:100 Alice`.
2. CLI отправляет gRPC‑вызов `Put(key="user:100", value="Alice")`.
3. Сервер генерирует монотонный `commitTS`, создаёт `MVCCKey` с инвертированной меткой.
4. Движок добавляет запись в WAL, затем в MemTable.
5. Воркер flush при достижении порога сбрасывает MemTable в SSTable уровня 0, а большие значения – в VLog.
6. Клиенту возвращается `commitTS`.

**Сценарий с ошибкой (конфликт транзакции):**
1. Транзакция A начинает `Begin`, получает `startTS = 10`.
2. Транзакция B начинает позже, тоже получает `startTS = 10` (если выполняется после A, её `startTS` мог быть больше, но допустим перехват).
3. Обе хотят изменить ключ `user:100`. A коммитит первой, её версия получает `commitTS = 11`.
4. B пытается закоммитить, но движок обнаруживает, что для `user:100` есть версия с `commitTS = 11 > startTS = 10`. Возвращается ошибка `ErrConflict`.
5. B может повторить транзакцию с новым `startTS`.

### 3.6 Обработка ошибок
- **Таймаут gRPC‑запроса** – клиент получает `DeadlineExceeded`; при необходимости повторяет вызов.
- **Конфликт транзакций** – возвращается `ErrConflict`, клиент перезапускает транзакцию.
- **Ошибка ввода/вывода (диск переполнен)** – `Put` возвращает `InternalError`, сервер логирует ошибку.
- **Данные не найдены** – `Get` возвращает `nil` и `found=false` (ошибка `ErrKeyNotFound` в embedded‑интерфейсе).

### 3.6.1 Формат ответа с ошибкой (gRPC и REST)
Ошибки передаются в формате:
```json
{
  "code": "CONFLICT",
  "message": "transaction conflict: key 'user:100' modified after start_ts",
  "details": {
    "key": "user:100",
    "start_ts": "10",
    "conflicting_ts": "11"
  }
}
```
HTTP‑статус: 409 Conflict. Для валидационных ошибок используется 400, для неавторизованных – 401, для несуществующего ключа – 404.

### 3.7 Масштабирование
- **Горизонтальное чтение:** В будущем можно добавлять Raft‑фолловеров, но в MVP – одна нода. Stateless gRPC‑серверов может быть несколько за балансировщиком (каждый запрос идёт к одному движку, т.к. данные локальны).
- **Отказоустойчивость:** Достигается в расширенном режиме через Raft.

### 3.7.1 Дополнительные аспекты production-масштабирования
- Консистентность при репликации будет обеспечена Raft‑группой.
- Шардирование (по диапазонам ключей) запланировано на следующий этап.
- Бэкапы – можно делать снапшот файлов SSTable и VLog.

### 🔹 Промпт 1. Вступительный: MANIFEST, VFS, Range Filter в SSTable

**Цель:** Доработать движок хранения ScoriaDB, добавив три критически важных компонента: **MANIFEST** (журнал метаданных SSTable), **VFS-абстракцию** (абстрактный слой файловой системы для тестируемости), и **диапазонный фильтр (min/max ключи)** в SSTable для ускорения сканирований.

**Что уже реализовано (текущее состояние проекта):**  
К настоящему моменту в проекте полностью реализованы и протестированы следующие пункты обязательного функционала (Раздел 2.1):
- ✅ **Пункт 1 (LSM‑движок, частично):**
  - MemTable на базе `google/btree` с корректным хранением ключей.
  - Value Log (WiscKey): значения > 64 байт выносятся в append‑only файл, доступный через `mmap` для zero‑copy чтения.
  - WAL с контрольными суммами CRC32 и механизмом `Recover()`, восстанавливающим MemTable после сбоя.
  - SSTable с блочным индексом, префиксным сжатием ключей и **Bloom‑фильтром** (пакет `internal/engine/sstable`).
  - Leveled Compaction (базовая реализация): движок способен выполнять слияние файлов при превышении порога в Level‑0.
- Все соответствующие unit‑тесты находятся в пакетах `internal/engine`, `internal/engine/sstable` и проходят.

**Что ещё не сделано (и почему мы берёмся именно за эти компоненты сейчас):**  
Несмотря на значительный прогресс, текущая реализация имеет несколько архитектурных недостатков, которые необходимо устранить перед добавлением MVCC, транзакций и API:

1.  **Отсутствие MANIFEST.** Сейчас список SSTable по уровням, вероятно, получается простым сканированием директории. Это ненадёжно: можно потерять файл, отсутствует атомарное обновление при flush/compaction, невозможно отследить историю изменений. MANIFEST (киллер‑фича 5, Раздел 6) решает эту проблему, предоставляя журнал `VersionEdit` с атомарным `fsync`, который позволяет восстановить консистентное состояние после любого сбоя.
2.  **Отсутствие VFS-абстракции.** Все файловые операции используют пакет `os` напрямую, что делает невозможным внедрение сбоев в тестах и усложняет сопровождение. VFS (Раздел 2.1, архитектурное решение) отделит логику движка от реальной файловой системы, позволив легко подменять реализацию в тестах.
3.  **Отсутствие диапазонного фильтра в SSTable.** Сейчас для сканирований используется только Bloom‑фильтр, который хорош для точечных запросов, но не отсекает файлы, не пересекающиеся с искомым диапазоном. Range Filter (min/max ключи в индексе) (Раздел 2.1) кратно ускорит операцию `Scan`, которая будет активно использоваться CLI и Web UI.

Эти три компонента являются фундаментом для дальнейшей разработки. Их внедрение сейчас гарантирует надёжность, тестируемость и производительность всех последующих слоёв.

**Требования к реализации (подробно):**

1.  **Создать пакет `internal/engine/manifest`:**
    - Определить структуру `VersionEdit` с полями: `NewFiles []SSTableInfo` (номер, уровень, диапазон ключей), `DeletedFiles []SSTableInfo`, `NextFileNum int`.
    - Реализовать `Manifest` как файл, открываемый через VFS. Каждая запись — JSON‑строка с полями `VersionEdit`, разделённая `\n`. После каждой записи выполняется `Sync()`.
    - Методы: `NewManifest(vfs VFS, path string)`, `Recover(vfs VFS, path string) ([]SSTableByLevel, error)`, `Apply(edit VersionEdit) error`.
    - Интегрировать `Manifest` в `LSMEngine`: добавить поле `manifest *manifest.Manifest`; при создании движка вызывать `Recover`; в методах `flushMemTable()` и `runCompaction()` после изменения состава файлов вызывать `Apply` и обновлять in‑memory список уровней.

2.  **Создать пакет `internal/engine/vfs`:**
    - Определить интерфейс `VFS` с методами `OpenFile`, `Write`, `Read`, `Sync`, `Rename`, `Delete` и т.д.
    - Реализовать `DefaultVFS`, делегирующую к `os`.
    - Внедрить VFS в `LSMEngineOptions`. Все существующие файловые операции (WAL, VLog, SSTable) должны использовать этот VFS вместо прямого вызова `os`. В конструкторах WAL, VLog, Reader/Writer SSTable добавить параметр `vfs VFS`.

3.  **Добавить диапазонный фильтр в SSTable:**
    - Модифицировать `Footer` в `sstable/encoding.go`: добавить поля `MinKey []byte`, `MaxKey []byte` и записывать их после основных полей.
    - В `Writer` при закрытии файла вычислить min/max ключи всех записанных блоков и сохранить их в Footer.
    - В `Reader` читать эти поля при открытии и сохранять в структуре `Reader`.
    - В методе `Get` / `Open` читателя перед чтением Bloom‑фильтра реализовать быструю проверку: если искомый ключ меньше `MinKey` или больше `MaxKey`, сразу возвращать `NotFound`.

4.  **Тесты (TDD):**
    - `TestManifestRecovery` – создать манифест, применить несколько правок, симулировать частичную запись и убедиться в корректном восстановлении до последнего консистентного состояния.

    - `TestVFSBasic` – проверить работу `DefaultVFS`.
    - Убедиться, что **все существующие тесты** продолжают проходить.

**Ожидаемый результат:** Движок хранения обзаводится надёжным журналом метаданных, абстрактным файловым слоем и эффективным пропуском нерелевантных файлов при сканировании. Все тесты зелёные. Проект готов к добавлению MVCC и транзакций.

### 🔹 Промпт 2. MVCC, транзакции и WriteBatch

**Цель:** Реализовать многоуровневую версионность (MVCC) с изоляцией Snapshot Isolation, атомарные пакетные операции (WriteBatch) и интерактивные транзакции с оптимистичной блокировкой.

**Что уже реализовано:**  
На предыдущем этапе (Промпт 1) мы завершили базовый движок хранения. Теперь у нас есть:
- Надёжное управление файлами через MANIFEST.
- Абстрактная файловая система (VFS), что упростит тестирование.
- Диапазонные фильтры в SSTable.
- Полный core‑LSM с WAL, MemTable, VLog, компакшеном.

До этого в проекте не было поддержки MVCC, транзакций или какой-либо версионности ключей. Настало время добавить их, опираясь на готовый и стабильный движок.

**Ссылки на ТЗ (обязательно перечитай перед началом):**
- **Раздел 1.3** – интерпретация пунктов «MVCC и Snapshot Isolation» и «Транзакции: WriteBatch и интерактивные».
- **Раздел 2.1 (пункты 2, 3, 4)** – MVCC, WriteBatch, Интерактивные транзакции входят в MVP.
- **Раздел 2.3 (Киллер‑фичи) – Фича 2** «Транзакции с оптимистичной блокировкой на основе временных меток».
- **Раздел 6 (Фича 2)** – пошаговое техническое объяснение работы оптимистичной блокировки.
- **Раздел 3.6.1** – формат ошибок.

**Требования к реализации (подробно):**

1.  **MVCC‑слой (`internal/mvcc`)**
    - **Структура `MVCCKey`:** `UserKey []byte` + `Timestamp uint64` (инвертированный, `math.MaxUint64 - commitTS`).
    - **Метод `Less`:** сравнивает сначала `UserKey` лексикографически; если равны — возвращает `k.Timestamp > other.Timestamp` (чтобы новые версии с меньшим `Timestamp` шли первыми).
    - **В MemTable и SSTable** добавляется флаг `Tombstone` (для удалённых ключей).
    - **Методы движка:** `PutWithTS(key, value []byte, commitTS uint64) error`, `GetWithTS(key []byte, snapshotTS uint64) ([]byte, error)`, `DeleteWithTS(key []byte, commitTS uint64) error`.
    - **Алгоритм `GetWithTS`**: ищет в MemTable (используя `DescendLessOrEqual` или перебором всех версий ключа), затем в SSTable, останавливаясь на первой видимой версии (с `commitTS <= snapshotTS`), не являющейся tombstone.

2.  **WriteBatch (`internal/txn`)**
    - Структура `WriteBatch` с методами `AddPut`, `AddDelete`.
    - `ApplyBatch(db *LSMEngine, batch *WriteBatch) (uint64, error)`: генерирует единый `commitTS`, сериализует все операции в WAL, применяет их к MemTable, возвращает `commitTS`. При сбое внутри — откатывает частичные изменения.

3.  **Интерактивная транзакция (`Transaction`)**
    - Создаётся через `Begin()`, сохраняет `startTS` (текущий максимальный `commitTS`).
    - Локальный буфер `writes map[string]*operation` для незакоммитченных изменений.
    - `txn.Get(key)` – сначала ищет в буфере, затем вызывает `GetWithTS(key, startTS)`.
    - `txn.Put` / `txn.Delete` – добавляют в буфер.
    - `txn.Commit()` – проверяет конфликты (для каждого ключа смотрит, не появилась ли версия с `commitTS > startTS`), атомарно фиксирует все изменения через `ApplyBatch`. При конфликте возвращает `ErrConflict`.
    - `txn.Rollback()` – очищает буфер.

4.  **Тесты (TDD)**
    - `TestMVCCPutGetMultipleVersions`
    - `TestMVCCDeleteTombstone`
    - `TestWriteBatchAtomicity`
    - `TestInteractiveCommitSuccess`
    - `TestInteractiveConflict`
    - `TestMVCCAcrossFlush`

**Ожидаемый результат:** Полноценная поддержка многоверсионности, Snapshot Isolation, атомарных батчей и интерактивных транзакций с обнаружением конфликтов. Все тесты проходят.

### 🔹 Промпт 3. Column Families (колоночные семейства)

- **Раздел 1.3** – интерпретация пункта «Column Families»: скрытое требование – атомарность при записи в несколько CF.
- **Раздел 2.1 (пункт 5)** – Column Families входят в обязательный функционал MVP.
- **Раздел 2.3 (Киллер‑фича 5)** – Column Families с атомарным WriteBatch.
- **Раздел 6 (Фича 5)** – детальное объяснение реализации.
- **Раздел 3.2 (Компоненты бэкенда)** – компонента Column Families.
- **Раздел 3.4 (Базы данных)** – системные CF `__auth__`, `__meta__` и пр.
- **Раздел 3.5 (Сценарий работы)** – успешный сценарий записи через CLI упоминает CF.
- **Раздел 3.6.1 (Формат ответа с ошибкой)** – пригодится для обработки ошибок.

**Текущее состояние проекта:**  
После Промпта 2 у нас есть работающий MVCC, WriteBatch и интерактивные транзакции. Движок представлен одной структурой `LSMEngine`, которая управляет одним LSM‑деревом (одним набором MemTable, SSTable, WAL). Все операции выполняются в этом единственном пространстве ключей.

**Требования к реализации:**

1. **Создать менеджер Column Families (`internal/cf`):**

     - `mu sync.RWMutex`
     - `cfs map[string]*LSMEngine` (сопоставление имени CF → движок)
   - Методы:
     - `NewRegistry(defaultOpts LSMEngineOptions) *Registry` – создаёт реестр и сразу добавляет CF `"default"` с указанными опциями.
     - `CreateCF(name string, opts LSMEngineOptions) error` – создаёт новый CF (отдельный экземпляр `LSMEngine`) и сохраняет в карте. Вернуть ошибку, если CF уже существует.
     - `GetCF(name string) (*LSMEngine, error)` – возвращает движок для заданного CF.
     - `DropCF(name string) error` – удаляет CF (освобождает ресурсы). Запрещено удалять системные CF (`__auth__`, `__meta__`).
     - `ListCFs() []string` – возвращает список имён CF.

2. **Адаптировать `LSMEngine` для работы с CF:**
   - В опции `LSMEngineOptions` добавить поле `Name string` (имя CF).
   - Каждый экземпляр `LSMEngine` самостоятельно управляет своими данными (WAL, VLog, SSTable, MANIFEST) в отдельной подпапке, определяемой именем CF. Например, для `default` путь будет `{workdir}/default/`, для `__auth__` – `{workdir}/__auth__/`. Организовать это в конструкторе `Open` движка.

3. **Обновить `WriteBatch` для поддержки нескольких CF:**
   - В структуре `WriteBatch` каждая операция должна содержать поле `CF string`.
   - При вызове `AddPut(cf, key, value)` или `AddDelete(cf, key)` указывать имя CF.
   - Метод `ApplyBatch` (в `LSMEngine` или в новом менеджере) должен:
     - Сгруппировать операции по CF.
     - Для каждого CF получить соответствующий движок через реестр.
     - **Атомарно** применить операции в каждом CF: для каждого CF внутри одной WAL‑транзакции (или общего WAL) записать изменения и обновить MemTable.

## 📁 Раздел 6. Объяснение каждой киллер-фичи (детальное)

### Фича 1: Zero‑copy Value Log с гибридным хранением (WiscKey + mmap)

**Как работает технически:**  
При `Put` значение длиннее 64 байт записывается в конец Value Log. В MemTable сохраняется только указатель (смещение + размер). Файл Value Log отображён в память через `syscall.Mmap`. При `Get` движок читает указатель и возвращает слайс, указывающий прямо в mmap‑область, без выделения новой памяти и копирования. Для проверки целостности перед значением в VLog хранится его CRC32.  
**Почему круто:** Исключаются лишние аллокации, ускоряется чтение больших значений (документы, блобы), само LSM‑дерево остаётся компактным.  
**Реализация:** `internal/engine/vlog.go` (запись, mmap, чтение с проверкой CRC).

### Фича 2: Транзакции с Snapshot Isolation и оптимистичной блокировкой

**Как работает технически:**  
`Transaction` при `Begin` запоминает `startTS`. Чтения (`Get`) внутри транзакции не видят изменения с `commitTS > startTS`. Свои изменения накапливаются в локальном буфере. При `Commit()` генерируется новый `commitTS` и для каждого изменённого ключа проверяется, не была ли записана новая версия после `startTS`. Если конфликта нет, все изменения атомарно фиксируются через `WriteBatch`.  
**Почему круто:** Позволяет выполнять сложные ACID‑операции без долгих блокировок. При конфликте клиент просто повторяет транзакцию.  
**Реализация:** `internal/txn/transaction.go`, интеграция с `internal/mvcc`.

### Фича 3: Встроенный Web UI с live‑просмотром и консолью администратора

**Как работает технически:**  
React‑приложение, собираемое Vite, встраивается в бинарник через `embed.FS`. При запуске сервера (порт 8080) отдаётся статика SPA, которая через REST API общается с движком. WebSocket‑соединение (опционально) позволяет получать мгновенные уведомления об изменениях.  
**Почему круто:** Администрирование БД возможно из браузера, без установки отдельного инструмента.  
**Реализация:** `web/` (React), `internal/api/rest` и `internal/api/ws`.

### Фича 4: CLI с офлайн‑инспекцией и AI‑ассистентом

**Как работает технически:**  
CLI на базе Cobra подключается к серверу по gRPC. Команда `scoria inspect` напрямую читает файлы БД (MANIFEST, SSTable) без запущенного сервера и выводит статистику. Команда `scoria ask` (опционально) преобразует естественный язык в JSON‑команду через локальный LLM (Ollama) и выполняет её.  
**Почему круто:** Быстрая диагностика без сервера и возможность задавать вопросы на ходу.  
**Реализация:** `cmd/cli`, `internal/engine` (inspect‑режим).

### Фича 5: MANIFEST и надёжное восстановление после сбоев

**Как работает технически:**  
Все изменения состава SSTable записываются в `MANIFEST`‑файл как последовательные `VersionEdit` с обязательным `fsync`. При старте манифест читается, и движок восстанавливает актуальное состояние уровней. Периодически создаётся снапшот манифеста для ускорения.  
**Почему круто:** Исключает потерю метаданных при внезапном отключении питания, делает восстановление детерминированным и быстрым.  
**Реализация:** `internal/engine/manifest`.
