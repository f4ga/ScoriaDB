## 📖 Для чего ScoriaDB?

**ScoriaDB** — это **встраиваемая** (embedded) база данных «ключ‑значение», написанная на чистом Go. Она создана для разработчиков, которым нужно:

- **Хранить данные локально** прямо в процессе своего приложения без отдельного сервера.
- **Быстро читать и писать** даже при больших объёмах данных (миллионы ключей).
- **Не терять данные** при внезапных сбоях.
- **Работать с большими значениями** (JSON, бинарные файлы) без раздувания оперативной памяти.
- **Использовать транзакции** для атомарных операций над несколькими ключами.
- **Иметь удобный интерфейс** как из кода, так и через командную строку или браузер.

Мы строим ScoriaDB, чтобы объединить скорость RocksDB, простоту BoltDB и удобство современных сетевых СУБД в одном бинарнике.

---

## 🤔 Когда стоит выбрать ScoriaDB?

- **Микросервисы и монолиты на Go** – когда нужна быстрая и надёжная БД «прямо в процессе».
- **IoT и Edge-устройства** – где нет возможности развернуть отдельную СУБД.
- **Обработка потоков данных** – временные ряды, счётчики, логи.
- **Промежуточный кэш с персистентностью** – чтобы не терять кэш при перезапуске.
- **Учебные и исследовательские проекты** – как пример промышленного системного программирования на Go.

ScoriaDB не подойдёт, если вам нужны сложные SQL-запросы или жёсткая распределённость «из коробки» (но мы планируем добавить Raft в будущем).

## ✨ Почему ScoriaDB?

### 1. Гибридное хранение (WiscKey + mmap)
Большие значения не засоряют LSM‑дерево. Они хранятся отдельно в журнале значений, отображённом в память, и читаются без лишнего копирования. Маленькие значения живут прямо в дереве. Это даёт **высокую производительность** и **экономит память**.

### 2. Транзакции с изоляцией снапшота
Мы поддерживаем интерактивные ACID‑транзакции с оптимистичной блокировкой. Вы можете выполнить несколько операций атомарно, не блокируя читателей.

### 3. Встраиваемость и простота
ScoriaDB компилируется в ваш бинарник как обычная Go‑библиотека. Никаких внешних зависимостей, портов или сложной настройки.

### 4. Удобные интерфейсы
- **Embedded API** – чистый Go‑код.
- **CLI** – управление через терминал.
- **gRPC API** – доступ из любого языка.
- **Web UI** – просмотр данных и администрирование через браузер.

### 5. Column Families
Логически разделяйте данные внутри одного хранилища с разными настройками компакшена, сжатия и TTL.

### 6. Надёжное восстановление после сбоев (Manifest)
Все изменения списка файлов журналируются в Manifest — аналог WAL для метаданных. После перезапуска движок восстанавливает консистентное состояние, а не сканирует директорию.

### 7. Открытый исходный код и активная разработка
Проект создаётся как демонстрация глубоких знаний системного программирования и будет развиваться с добавлением распределённого режима, time‑travel запросов и многого другого.

## 🧠 Как это работает (простыми словами)

1. **Запись**: вы вызываете `Put("user:1", "Alice")`. Данные сначала попадают в журнал (WAL) на диск, а потом в быструю структуру в памяти — MemTable.
2. **Чтение**: вы вызываете `Get("user:1")`. Мы сначала ищем в памяти, а если там нет — поднимаем данные с диска, но только нужный кусочек, не читая всё подряд.
3. **Сброс на диск**: когда память заполняется, мы аккуратно переносим данные в файлы SSTable — отсортированные и сжатые.
4. **Уборка**: старые версии ключей и мусор автоматически удаляются, чтобы база не разрасталась бесконечно.
5. **Большие значения**: если значение очень длинное, мы храним его отдельно в специальном «журнале значений» и показываем напрямую из памяти, не копируя.

Более детально архитектура и внутренние решения описаны [ниже](#архитектура).

## 🚀 Быстрый запуск

### Сборка из исходников

```bash
git clone https://github.com/your-org/scoriadb.git
cd scoriadb
go build ./...
```

### Запуск сервера

```bash
go run cmd/server/main.go
```

Сервер слушает gRPC на порту **50051** и HTTP/Web UI на порту **8080**.

### Установка CLI

```bash
go run cmd/cli/main.go --help
```

### Docker

```bash
docker-compose up -d
```

Web UI откроется по адресу http://localhost:8080.

## 💡 Как пользоваться в своём коде

```go
package main

import (
    "fmt"
    "log"

    "github.com/your-org/scoriadb/pkg/scoria"
)

func main() {
    db, err := scoria.Open(scoria.DefaultOptions("/tmp/mydb"))
    if err != nil {
        log.Fatal(err)
    }
    defer db.Close()

    _ = db.Put([]byte("привет"), []byte("мир"))
    val, _ := db.Get([]byte("привет"))
    fmt.Println(string(val)) // "мир"
}
```

## 🧩 Ключевые возможности (MVP)

> **Легенда:** ✅ – уже реализовано, 🔜 – в активной разработке

- ✅ **LSM‑дерево** — MemTable (B‑tree), Value Log, WAL с CRC
- ✅ **SSTable** — блочный индекс, префиксное сжатие, фильтр Блума, диапазонный фильтр (min/max)
- ✅ **Manifest** — журнал метаданных для надёжного восстановления
- ✅ **VFS‑абстракция** — подменяемая файловая система для тестирования
- ✅ **Leveled Compaction** — базовое слияние уровней
- 🔜 **MVCC** — изоляция Snapshot Isolation, версионность ключей (следующий этап)
- 🔜 **Транзакции** — интерактивные (Begin/Commit/Rollback) и пакетный WriteBatch
- 🔜 **Column Families** — независимые LSM‑деревья с общими настройками
- 🔜 **Embedded Go API** (`pkg/scoria`) — чистый интерфейс для встраивания
- 🔜 **gRPC API** — клиент‑серверный доступ со стримингом для Scan
- 🔜 **CLI‑клиент** (`scoria`) — управление данными, офлайн‑инспекция
- 🔜 **Web UI** (React) — просмотр ключей, администрирование пользователей
- 🔜 **Аутентификация и авторизация** — JWT, bcrypt, роли (admin, readwrite, readonly)
- 🔜 **Prometheus‑метрики и health‑check**
- 🔜 **Контейнеризация** — Docker, docker‑compose

## 📁 Архитектура

```
scoriadb/                       # Go-модуль (embed)
├── pkg/scoria/                 # публичный интерфейс DB
├── internal/
│   ├── engine/                 # LSM-ядро: MemTable, SSTable, Value Log, WAL, Manifest
│   │   ├── sstable/            # чтение/запись SSTable, Bloom, range filter
│   │   ├── manifest/           # VersionEdit log
│   │   └── vfs/                # абстракция файловой системы
│   ├── mvcc/                   # кодирование/декодирование версий (в разработке)
│   ├── txn/                    # транзакции: интерактивные, batch (в разработке)
│   ├── cf/                     # реестр Column Families (в разработке)
│   ├── auth/                   # пользователи, JWT (в разработке)
│   └── api/
│       ├── grpc/               # gRPC-сервер (в разработке)
│       └── rest/               # REST-прокси (в разработке)
├── cmd/
│   ├── server/                 # бинарник сервера
│   └── cli/                    # CLI-клиент
├── web/                        # React SPA (встраивается через embed)
└── deployments/                # Dockerfile, docker-compose
```

Движок разделён на **исполняющую плоскость** (`engine`) и **плоскость управления**, которая появится в распределённой версии. Благодаря этому одну и ту же библиотеку можно использовать локально или в составе Raft‑кластера без изменений.

## 📝 Пояснение отдельных решений

- **Manifest вместо сканирования директории.** Все изменения состава файлов пишутся в append‑only лог с `fsync`. При перезапуске читается манифест, восстанавливается актуальное состояние. Это исключает потерю метаданных при внезапном выключении и упрощает отладку.
- **Диапазонный фильтр min/max.** Помимо фильтра Блума, каждый SSTable хранит минимальный и максимальный ключ. При `Scan` быстро отсекаются файлы, не пересекающиеся с искомым диапазоном — сокращается количество операций ввода‑вывода.
- **VFS-абстракция.** В продакшене используется обычный `os.File`, но в тестах реализация подменяется, чтобы симулировать ошибки записи, нехватку места, сбои `fsync`. Это повышает покрытие краевых случаев без правки самого движка.
- **Value Log с zero‑copy.** Файл значений отображается в память через `mmap`; при чтении возвращается слайс, указывающий прямо на память, минуя лишние аллокации. Контрольная сумма, хранимая перед значением, проверяется перед возвратом.
- **Встроенный Web UI.** React‑приложение компилируется в статику и встраивается в бинарник через `embed.FS`. При запуске сервера интерфейс доступен на том же порту — отдельный веб‑сервер не требуется.
- **CLI с офлайн‑инспекцией.** Утилита `scoria inspect` читает директорию с данными остановленного экземпляра и выводит статистику: количество ключей, фрагментацию Value Log, размеры уровней — без запуска движка.
