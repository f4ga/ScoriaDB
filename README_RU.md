## 📖 Что такое ScoriaDB?

**ScoriaDB** — это встраиваемая key‑value база данных на чистом Go.  
Внутри — LSM‑дерево, MVCC, Column Families, WAL и Value Log в стиле WiscKey.


- **Как библиотека** — вы пишете `import _ "github.com/f4ga/scoriadb/pkg/scoria"` и получаете LSM‑движок внутри своего процесса. Никаких внешних зависимостей, никакого cgo.
- **Как сервер** — запускаете один бинарник, и он поднимает gRPC, REST, WebSocket и CLI. Готовый бэкенд для микросервисов на любом языке.

> Проект сейчас на финишной прямой к **v0.1.0**. Почти всё работает, тесты зелёные.

## ✨ Почему ScoriaDB?

| Преимущество | Что даёт на практике |
| :--- | :--- |
| **Встраиваемость** | Чистый Go. Не нужно `apt-get install rocksdb` и думать о cgo. |
| **Готовый сервер** | gRPC, REST, CLI, WebSocket — из коробки. Не нужно писать HTTP‑обёртки, достаточно запустить `scoria-server`. |
| **ACID‑транзакции** | Snapshot Isolation, интерактивные транзакции для сложных сценариев, атомарные WriteBatch для пачек операций. |
| **Column Families** | Независимые LSM‑деревья: можно хранить метаданные отдельно от данных, настраивать компакшн под каждый CF. |
| **MVCC** | Читатели не блокируют писателей.( Но конкурентность еще не идеальна)|
| **Кросс‑языковой доступ** | gRPC — клиенты для 12+ языков. Ваши Python‑скрипты и Java‑микросервисы спокойно ходят в одну базу. |
| **Надёжность** | WAL с fsync, Manifest с fsync, CRC32 на каждый блок. После `kill -9` данные не теряются. |
| **Производительность** | ~150 нс на чтение, ~1 мкс на запись (маленькие ключи)1 |

## 📊 Бенчмарки

*Тестовая машина: Intel Core i3-1215U (8 потоков), 16GB RAM, NVMe SSD, Go 1.23+, Linux amd64.*  
*Запуск: `go test -bench=. -count=5 ./internal/engine ./pkg/scoria | benchstat`*

| Операция | Размер значения | Время (нс/оп) | Время (мс/оп) | Пропускная способность |
|----------|----------------|---------------|---------------|------------------------|
| `engine.Put` (маленькое) | 16 байт | **1 070** | 0.00107 | ~935 000 оп/с |
| `engine.Put` (большое) | 4 КБ (попадает в VLog) | **4 785** | 0.00479 | ~209 000 оп/с |
| `engine.Get` (попадание) | ключ в MemTable | **152** | 0.00015 | ~6 580 000 оп/с |
| `engine.Get` (промах) | ключ не существует | **310** | 0.00031 | ~3 225 000 оп/с |
| `ScoriaDB.Put` (публичное API) | 16 байт | **1 063** | 0.00106 | ~940 000 оп/с |
| `ScoriaDB.Get` (публичное API) | 16 байт | **144** | 0.00014 | ~6 940 000 оп/с |

**Пояснения:**
- **engine.Put** — прямой вызов метода движка, без оверхеда API.
- **ScoriaDB.Put** — через публичный интерфейс `DB.Put()`. Накладные расходы <1% — практически прозрачно.
- **Большие значения** (>64 байт) уходят в Value Log. Чтение такого значения включает проверку CRC32 и копирование из mmap (да, пока не zero‑copy, честно).
- **Промах** быстрее попадания, потому что не нужно распаковывать значение.

**Что означают эти цифры в реальности:**
- Если вы пишете пачками через WriteBatch (скажем, по 100 операций), то накладные расходы fsync амортизируются, и вы упираетесь в диск. На NVMe — десятки микросекунд на запись.
- Чтение почти никогда не тормозит, даже при активной записи — MVCC рулит.

## 📊 Сравнение с Redis

**Важное предупреждение:** ScoriaDB **не** является заменой Redis. Они решают разные задачи. Redis — это супербыстрый in‑memory кеш с сетевым доступом. ScoriaDB — это дисковое, встраиваемое хранилище с транзакциями. Сравнение ниже — чисто для понимания цифр.

| Характеристика | ScoriaDB (встраиваемая) | Redis CE (сетевая) |
| :--- | :--- | :--- |
| **Развёртывание** | Библиотека или сервер | Только сервер |
| **Сетевые накладные расходы** | Нет (при embedded) | TCP (обычно 0,1–0,2 мс) |
| **Задержка чтения (Get)** | ~150 нс | ~0,24–0,31 мс |
| **Задержка записи (Set)** | ~1 070 нс | ~0,45 мс |
| **Персистентность** | Полная, с fsync | Опциональная (RDB/AOF) |
| **Транзакции** | ACID, Snapshot Isolation, блокировка оптимистичная | Нет (только pipelining) |
| **MVCC** | Да, читатели не блокируют писателей | Нет |
| **Column Families** | Да | Нет |
| **Встраиваемость** | Да, `import` | Нет, отдельный процесс |
| **Языки** | gRPC (12+), плюс прямой Go API | Нативные клиенты под каждый язык |

**Итог:** если вам нужен суперскоростной кеш с кучей структур данных — берите Redis. Если нужна надёжная, встраиваемая база с транзакциями и возможностью удалённого доступа — попробуйте ScoriaDB.

## 🧩 Возможности и функции

### 1. LSM‑движок

| Компонент | Статус | Пояснение |
| :--- | :---: | :--- |
| **MemTable** | ✅ | B‑tree из `google/btree` с глобальным мьютексом (да, это узкое место, но работает). |
| **SSTable** | ✅ | Блочный индекс, префиксное сжатие ключей, фильтр Блума, диапазонный фильтр (min/max key). |
| **Leveled Compaction** | ✅ | Классический алгоритм из LevelDB: L0 → L1 → L2 → L3. L0 могут пересекаться, L1+ — нет. |
| **Сжатие блоков** | ✅ | Snappy и Zstd на уровне блоков SSTable. |
| **Bloom‑фильтр** | ✅ | Вероятностная структура, чтобы не читать SSTable, если ключа точно нет. |
| **Value Log (WiscKey)** | ✅ | Значения >64 байт выносятся в отдельный append‑only файл `.vlog`. Чтение через mmap (пока с копированием). |

### 2. Надёжность и журналы

| Компонент | Статус | Пояснение |
| :--- | :---: | :--- |
| **WAL** | ✅ | Каждая операция пишется в журнал с CRC32 перед тем, как попасть в MemTable. Восстановление после краша. |
| **Manifest** | ✅ | JSON‑журнал изменений набора SSTable. Каждая запись fsync’ится. |
| **fsync на каждый батч** | ✅ | Да, это дорого, но данные не теряются. |
| **CRC32 блоков** | ✅ | Каждый блок SSTable и Value Log имеет контрольную сумму. |
| **Fail‑safe VLog** | ✅ | При повреждении magic‑числа файл переименовывается в `.corrupt`, создаётся новый, данные восстанавливаются из WAL. |

### 3. Транзакции и MVCC

| Возможность | Статус | Пояснение |
| :--- | :---: | :--- |
| **MVCC** | ✅ | Каждая запись получает `commitTS`. Удаление — tombstone. |
| **Snapshot Isolation** | ✅ | Транзакция видит консистентный срез на момент `startTS`. |
| **Интерактивные транзакции** | ✅ | `Begin()` → `Get`/`Put`/`Delete` → `Commit()`/`Rollback()`. Повтор при конфликте. |
| **WriteBatch** | ✅ | Атомарная группа операций — либо все, либо ничего. |
| **Конфликт‑чек** | ✅ | При коммите проверяется, не изменился ли ключ после `startTS`. |
| **Генератор меток** | ✅ | Атомарный `atomic.AddUint64`, персистентный (сохраняется в Manifest). |

### 4. Column Families

| Возможность | Статус | Пояснение |
| :--- | :---: | :--- |
| **Независимые LSM** | ✅ | У каждого CF свой набор файлов, своя MemTable, свой компакшн. |
| **Общий WAL** | ✅ | Атомарные WriteBatch между CF — одна запись в WAL. |
| **Параметры на CF** | 🛠️ | Пока только размер MemTable (в планах — отдельные настройки компакшена, TTL). |

### 5. API и интерфейсы

| Интерфейс | Статус | Пояснение |
| :--- | :---: | :--- |
| **Embedded Go API** | ✅ | `type DB interface { Get, Put, Delete, Scan, Begin, ... }`. |
| **gRPC API** | ✅ | Протофайл в `proto/`, сгенерированный код в `scoriadb/`. Поддержка стриминга Scan. |
| **REST API** | ✅ | Gin‑сервер: `GET /api/v1/kv/{key}`, `POST /api/v1/kv`, `DELETE /...`. Swagger пока нет, но будет. |
| **WebSocket** | ✅ | Подписка на изменения ключей: клиент получает нотификацию при `Put`/`Delete`. |
| **CLI (`scoria`)** | ✅ | Cobra: `set`, `get`, `del`, `scan`, `txn begin/commit`, `admin auth`, `admin user create`. Интерактивный режим. |

### 6. Безопасность и мониторинг

| Возможность | Статус | Пояснение |
| :--- | :---: | :--- |
| **JWT‑аутентификация** | ✅ | Токены с ролями, подпись HMAC или RSA. |
| **Роли** | ✅ | `admin` (всё), `readwrite` (CRUD), `readonly` (только чтение). |
| **Seed‑пользователь** | ✅ | При первом запуске создаётся `admin/admin` — смените пароль! |
| **Prometheus метрики** | ✅ | Эндпоинт `/metrics`: счётчики операций, гистограммы задержек, размеры MemTable, количество файлов на уровнях. |
| **Health / ready** | ✅ | `/health` — всегда 200, `/ready` — проверяет, что можно читать. |

## 🛡️ Надёжность и восстановление

ScoriaDB спроектирована так, чтобы **не терять данные** даже при внезапном отключении питания.

### Как это работает:

1. **WAL (Write‑Ahead Log).**  
   Перед тем как положить операцию в MemTable, я пишу её в WAL вместе с CRC32. И вызываю `fsync`. Если сервер упадёт в этот момент — всё равно операция либо целиком в WAL, либо нет. При старте я прохожу WAL и восстанавливаю всё, что не успело попасть в SSTable.

2. **Manifest (журнал метаданных).**  
   При компакшене я удаляю старые SSTable и добавляю новые. Вместо того чтобы сканировать папку (ненадёжно), я пишу каждое изменение в Manifest и тоже `fsync`. При старте читаю Manifest — и точно знаю, какие файлы сейчас активны.

3. **Value Log.**  
   Каждый `.vlog` имеет заголовок с магическим числом. При открытии я проверяю их. Если не совпадает, следовательно файл повреждён. Переименовываю в `.corrupt`, создаю новый, а данные восстанавливаю из WAL. Записи, указывающие на старый файл, пропускаются (они потеряны), но база остаётся консистентной.

### Что это даёт:

- **Фиксация записи** происходит в момент, когда `Put` вернул успех. Если после этого наступил `kill -9` — запись не пропадёт.
- **Метаданные** никогда не расходятся с реальными файлами.
- Никакой «ручной починки» после сбоя — база сама восстанавливается.

### Цена:

- `fsync` на каждый WriteBatch — это дорого. Примерно **в 5 раз медленнее**, чем запись в буфер. В v0.2.0 сделаем Group Commit, чтобы амортизировать накладные расходы.

## 🕰️ Как работает MVCC

MVCC (Multi‑Version Concurrency Control) — это способ одновременно читать и писать без блокировок.

### Основная идея:

- Каждая запись (Put) создаёт **новую версию** ключа, а не затирает старую.
- У каждой версии есть временная метка `commitTS` (uint64).
- Транзакция при старте получает `startTS` — это номер текущего момента.
- Внутри транзакции все чтения видят версии с `commitTS ≤ startTS`.
- При коммите генерируется новый `commitTS` (больше всех предыдущих). Если какой‑то ключ, который транзакция изменяла, уже имеет версию с `commitTS > startTS` — значит, кто‑то успел записать его после начала транзакции. Конфликт! Транзакция откатывается, и вы должны повторить её.

### Инвертированная метка — почему свежие записи в начале?

Я храню ключи в LSM в формате: `[user_key][^commitTS]`.  
Где `^commitTS` — это побитовое NOT от метки. Так как `commitTS` растёт, `^commitTS` убывает. Поэтому свежие записи оказываются **раньше** в лексикографическом порядке. Итератор по умолчанию идёт от новых к старым — очень удобно для сценариев, где нужна последняя версия.

### Пример:

```go
db.Put("user:1", "alice")   // commitTS = 100
db.Put("user:1", "bob")     // commitTS = 101

// Итератор по "user:1" покажет сначала bob (commitTS 101), потом alice (100).
```

### Читатели не блокируют писателей — как так?

- Писатели добавляют новые версии, не трогая старые.
- Читатели работают со снапшотом (`startTS`), который зафиксирован в начале.
- Никаких блокировок на чтение. Вообще.

Цена: старые версии накапливаются. Их удаляет компакшн, но он не трогает версии, которые ещё нужны активным транзакциям.

## 🚀 Быстрый старт

### 1. Через Docker (рекомендуется)

```bash
git clone https://github.com/f4ga/ScoriaDB.git
cd ScoriaDB
docker compose -f deployments/docker-compose.yml up --build
```

Сервер запустится на:
- `:50051` — gRPC
- `:8080` — REST и WebSocket (пока нет Web UI, только API)

Проверить:

```bash
# В другом терминале
docker exec -it scoria-server ./scoria-cli admin auth admin admin
# Получите токен, затем:
docker exec -it scoria-server ./scoria-cli --token <токен> set hello world
docker exec -it scoria-server ./scoria-cli --token <токен> get hello
```

### 2. Локальная сборка

```bash
go build -o scoria-server ./cmd/server
go build -o scoria-cli ./cmd/cli

./scoria-server &
# Сервер слушает :50051 и :8080

# Получить токен (при первом запуске создаётся admin/admin)
TOKEN=$(./scoria-cli admin auth admin admin)

# CRUD
./scoria-cli --token "$TOKEN" set hello world
./scoria-cli --token "$TOKEN" get hello

# Интерактивная транзакция
./scoria-cli --token "$TOKEN" txn begin
./scoria-cli --token "$TOKEN" txn set a 1
./scoria-cli --token "$TOKEN" txn set b 2
./scoria-cli --token "$TOKEN" txn commit

# Скан
./scoria-cli --token "$TOKEN" scan -p "a" -l 10

# Запустить интерактивный режим CLI
./scoria-cli --token "$TOKEN" shell
```

### 3. Из кода Go (встраиваемый режим)

```go
package main

import (
    "log"
    "github.com/f4ga/scoriadb/pkg/scoria"
)

func main() {
    db, err := scoria.Open("/path/to/data", nil)
    if err != nil {
        log.Fatal(err)
    }
    defer db.Close()

    // Простая запись
    db.Put([]byte("hello"), []byte("world"))

    // Транзакция
    txn := db.Begin()
    val, _ := txn.Get([]byte("hello"))
    txn.Put([]byte("foo"), []byte("bar"))
    if err := txn.Commit(); err != nil {
        log.Fatal(err)
    }
}
```

go test -bench=. ./internal/engine ./pkg/scoria

## ❓ FAQ

### 1. ScoriaDB готова к продакшену?
**v0.1.0 — да, для умеренных нагрузок и встраиваемых сценариев.** Если у вас 10–20 конкурентных записей и несколько читателей — полетит отлично. Если вы планируете 1000+ горутин, которые постоянно пишут — лучше подождать v0.3.0, где появится lock‑free skip list и автоматический GC.

### 2. Zero‑copy реально работает?
**Нет. Пока нет.** Текущая реализация копирует данные из mmap‑региона в новую память. Это сделано, чтобы избежать SIGSEGV при закрытии базы. Настоящий zero‑copy будет в v0.3.0.

### 3. Почему запись тормозит?
Из‑за `fsync` после каждого батча. Это гарантирует, что данные не потеряются. В v0.2.0 внедрим Group Commit — тогда задержки упадут в 3–5 раз, примерно до уровня буферизованной записи, но с сохранением надёжности.

### 4. Чем отличается от BadgerDB?
BadgerDB — хорошая библиотека, но в ней нет:
- встроенного сервера (gRPC/REST/CLI);
- интерактивных транзакций (`Begin/Commit`);
- Column Families.
ScoriaDB даёт всё это плюс чистый Go без cgo.

### 5. А от BoltDB?
BoltDB — B+‑дерево, один писатель, никакой сети. ScoriaDB — LSM, конкурентная запись, MVCC, есть сервер.

### 6. Можно ли использовать из Python?
Да, через gRPC. 

### 7. Как запустить Web UI?
Пока никак — он в разработке для v0.2.0. Следите за релизами.

### 8. Поддерживается ли распределённый режим?
Нет, это задача для v0.5.0. Пока только локальный режим.

### 9. Могу ли я внести свой вклад?
Да! Pull Request'ы приветствуются. Прочтите [CONTRIBUTING.md](CONTRIBUTING.md). Особенно нужна помощь с автоматическим GC и lock‑free структурами.
