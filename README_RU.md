## 📖 Что такое ScoriaDB?

**ScoriaDB** — это встраиваемая key‑value база данных, которая заменит использование связок из сетевого и встраиваемого хранилища.

- **Как встраиваемая библиотека** — компилируется прямо в ваш Go‑процесс, давая максимальную скорость без внешних зависимостей.
- **Как сервер** — предоставляет встроенные gRPC, REST и CLI (Web UI запланирован на v0.1.1), не требуя ни строчки дополнительного кода.

> Проект сейчас в цикле выпуска **v0.1.0**. Все ключевые компоненты реализованы и протестированы. В разработке находится демо‑инструмент **Scorix** для анализа логов.

---

## 👥 Для кого

ScoriaDB полезна:

- **Go‑разработчикам**, которые хотят добавить персистентное KV‑хранилище в свой микросервис, CLI‑утилиту или агент — без лишней инфраструктуры.
- **IoT и Edge‑инженерам** — когда нужно локальное хранилище на устройстве, но при этом важен удалённый доступ и мониторинг.
- **Разработчикам на Python, Java, C++ и других языках** — доступ через gRPC для любого, поддерживающего gRPC, языка.
- **Командам с микросервисной архитектурой** — один сервер, много клиентов на разных языках, единый API.
- **Всем, кому нужен простой персистентный KV‑движок**, который разворачивается за секунды с помощью Docker и предлагает встроенные интерфейсы CLI и REST/gRPC.

## 📊 Сравнение с Redis

ScoriaDB **не** является прямой заменой Redis. Redis — это удалённая, in‑memory платформа данных; ScoriaDB — встраиваемая, дисково‑ориентированная база.  
Таблица ниже показывает **локальные бенчмарки** (без сетевых накладных расходов) и **сетевые бенчмарки** Redis, чтобы проиллюстрировать архитектурную разницу.

| Характеристика | ScoriaDB (локально) | Redis CE (сетевые) |
| :--- | :--- | :--- |
| **Развёртывание** | Встраиваемая библиотека или отдельный сервер | Отдельный серверный процесс |
| **Сетевые накладные расходы** | Нет (embedded‑режим) | TCP (0,1–0,2 мс+) |
| **Задержка чтения (Get)** | ~150 нс | ~0,24–0,31 мс  |
| **Задержка записи (Set)** | ~1 070 нс | ~0,45 мс  |
| **Персистентность данных** | Полная (WAL, Manifest, fsync) | Опциональная (RDB, AOF) |
| **Транзакции** | ACID, Snapshot Isolation | Нет (pipelining) |
| **Поддержка языков** | gRPC (12+ языков) | Нативный протокол (клиентские библиотеки) |

YCSB бенчмарки будут опубликованы в версии v1.0.0.

## 🧩 Возможности и функции

### Storage Engine
| Функция | Описание | Статус |
| :--- | :--- | :--- |
| **LSM‑дерево** | Отсортированная MemTable (B‑tree) с периодическим сбросом в SSTable на диск. | ✅ |
| **WAL (Write‑Ahead Log)** | Каждая операция журналируется с контрольной суммой CRC32 перед попаданием в MemTable. | ✅ |
| **Value Log (WiscKey)** | Значения > 64 байт выносятся в отдельный append‑only файл; mmap для zero‑copy чтения. | ✅ |
| **SSTable** | Блочный индекс, префиксное сжатие ключей, фильтр Блума, диапазонный фильтр (min/max ключ). | ✅ |
| **Leveled Compaction** | Фоновое слияние SSTable для освобождения места и удаления tombstone. | ✅ |
| **Сжатие** | Snappy и Zstd на уровне блоков SSTable. | ✅ |
| **GC Value Log** | Сборщик мусора для Value Log. | 🛠️ v0.1.0 |
| **TTL для записей** | Автоматическое удаление записей. | ⏳ v1.0.0 |
| **Снапшот‑бэкап** | Мгновенные снапшоты через hardlink. | ⏳ v1.0.0 |

| Функция | Описание | Статус |

## 🛡️ Надёжность и восстановление

**Manifest** — это журнал метаданных, который записывает каждое изменение в наборе файлов с атомарным `fsync`. **WAL** сохраняет каждую операцию с CRC32. При старте движок читает Manifest, чтобы восстановить точную файловую структуру, и WAL, чтобы восстановить незафиксированные записи.

Вместе они гарантируют **полное восстановление после внезапного отключения питания** — без повреждения метаданных и без потери подтверждённых записей.

---

## 🕰️ Как работает MVCC

**MVCC (Multi‑Version Concurrency Control)** означает, что каждая запись создаёт новую версию ключа вместо перезаписи старой. Каждая версия несёт временную метку (`commitTS`).

1. При `Put` создаётся новая версия ключа с `commitTS = <текущее время>`.
2. Когда транзакция вызывает `Begin()`, она получает `startTS` — снапшот на этот момент.
3. Все чтения внутри транзакции видят только версии с `commitTS ≤ startTS`.
4. При `Commit()` движок проверяет, не был ли ключ изменён после `startTS` (обнаружение конфликтов).

**Почему это важно:**
- **Писатели никогда не блокируют читателей.**
- **Snapshot Isolation.** Консистентный взгляд на данные даже при конкурентной записи.
- **Time Travel станет возможным** (запланирован на v1.0.0).

## 🚀 Быстрый старт

### Docker (рекомендуется)

```bash
git clone https://github.com/f4ga/scoriadb.git && cd scoriadb
docker compose -f deployments/docker-compose.yml up --build
```

Запускает сервер (gRPC :50051, HTTP :8080) и выполняет интеграционный тест CLI.

### Локальная сборка

```bash
git clone https://github.com/f4ga/scoriadb.git && cd scoriadb
go build -o scoria-server ./cmd/server
go build -o scoria-cli ./cmd/cli

./scoria-server &   # сервер на :50051 и :8080

# Получить токен (при первом старте создаётся admin/admin)
TOKEN=$(./scoria-cli --addr localhost:50051 admin auth admin admin)

# CRUD
./scoria-cli --addr localhost:50051 --token "$TOKEN" set hello world
./scoria-cli --addr localhost:50051 --token "$TOKEN" get hello
```

### Конкурентная запись и чтение — безопасно благодаря MVCC

```go
var wg sync.WaitGroup
for i := 0; i < 10; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        txn, _ := db.Begin("default")
        txn.Put([]byte(fmt.Sprintf("key%d", id)), []byte(fmt.Sprintf("val%d", id)))
        txn.Commit()
    }(i)
}
wg.Wait()
// Все изменения изолированы, конфликты обработаны
```

### Запуск тестов (с гонкой) и бенчмарков

```bash
go test -race ./...
go test -bench=. ./internal/engine ./pkg/scoria
```

Все тесты ядра проходят с флагом `-race` — конкурентный доступ свободен от гонок.

## ❓ FAQ

### 1. ScoriaDB — это библиотека или сервер?
**И то, и другое.** Вы можете встроить её как Go‑библиотеку (`import "github.com/f4ga/scoriadb/pkg/scoria"`) или запустить как отдельный сервер с gRPC, REST и CLI.

### 2. Чем она отличается от BoltDB?
BoltDB — B+‑дерево, один писатель, нет сети. ScoriaDB — LSM, конкурентная запись, MVCC, встроенные gRPC/CLI.

### 3. Чем она отличается от BadgerDB?
Обе используют WiscKey. BadgerDB — только библиотека, без сервера, без интерактивных транзакций. ScoriaDB добавляет Column Families, MVCC, gRPC и CLI.

### 4. Чем она отличается от RocksDB?
RocksDB — C++ (cgo), нет встроенного сервера. ScoriaDB — чистый Go, с собственным gRPC/CLI и интерактивными ACID‑транзакциями.

### 5. Почему выбрана Snapshot Isolation, а не полная сериализуемость?
Snapshot Isolation даёт оптимальный баланс между производительностью и консистентностью для ключ‑значение нагрузок. Читатели не блокируются, писатели проверяют конфликты только при коммите. Полная сериализуемость потребовала бы более тяжёлых блокировок и не планируется в v1.0.

### 6. Могу ли я использовать ScoriaDB из Python/Java/C++?
Да, через gRPC API.

### 7. Поддерживает ли ScoriaDB SQL?
Нет. ScoriaDB — это KV‑база данных. SQL‑подобные запросы не планируются.

### 8. Что происходит с данными при внезапном отключении питания?
WAL + Manifest с fsync гарантируют полное восстановление без повреждения метаданных и без потери данных.

### 9. Готова ли она к продакшену?
v0.1.0 будет первым стабильным релизом. Для критичных систем мы рекомендуем дождаться v1.0.0.

### 10. Какие структуры данных сейчас поддерживаются?
Строки. v1.0.0 представит Sorted Sets, Lists, очереди и индексы.

### 11. Будет ли распределённый режим?
Да, в v1.0.0: Raft‑репликация, шардирование и федерация.

### 12. Почему gRPC, а не REST?
gRPC быстрее (HTTP/2, бинарный protobuf) и поддерживает стриминг. REST используется для Web UI и отладки.
